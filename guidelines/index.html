UI Goiás Guideline

UI Goiás Guideline

Diretrizes de design de interface de usuário para sistemas e apps do Estado de Goiás.

Objetivo

O presente documento tem como objetivo apresentar uma proposta de padrão de interface para as aplicações web do Governo do Estado de Goiás e suas secretarias.

Apresentação

A proposta deste trabalho é criar e implantar uma identidade visual única para aplicações web do estado de Goiás. Identidade visual, como o próprio nome diz, é o conjunto de elementos que compõem a apresentação visual de uma empresa, produto ou serviço. Uma identidade visual sólida é um item indispensável para que os usuários sintam se seguros de que estão interagindo em um ambiente confiável. Por isso é extremamente necessário que alguns itens estejam presentes em todas as páginas de uma aplicação. Sendo assim, o padrão proposto será apresentado, bem como algumas informações a respeito de usabilidade para aplicações web.

Experiência do Usuário

O estudo da experiência do usuário em uma aplicação web é importante para torna-la mais eficiente e produtiva.

Se um aplicativo possui uma interface de uso difícil, confusa, cansativa ou mal organizada, fatalmente o usuário não irá utilizala.

Cada minuto que o usuário gasta tentando encontrar uma função é um tempo perdido, precioso para que ele cumpra uma tarefa.

Para desenvolvermos uma usabilidade adequada, é necessário que se pense desde o início do projeto naquele que é o maior interessado em que o aplicativo seja simples e fácil: o usuário final.

Deve se conhecer o escopo, as limitações e os anseios do usuário desde o início do projeto, voltando os olhos não só para a robustez e a eficiência do sistema, mas também para as necessidades do usuário.

Uma aplicação de design voltada para o usuário, utilizando avaliações de usabilidade durante todo o ciclo de desenvolvimento, resulta em vários benefícios, como:

  • [Menores custos de desenvolvimento] - Muitos erros de usabilidade são devidos a uma má concepção da mesma.
  • [Redução do tempo de projeto] - A correção de erros em fases iniciais do projeto, diminui a necessidade de um retrabalho em relação ao desenvolvimento, o que acarreta uma diminuição no tempo do projeto.
  • [Menores custos de manutenção e suporte] - Um aplicativo fácil de aprender e usar resulta em redução de custos de aprendizagem e de suporte. Basicamente, uma aplicação satisfatória preenche os seguintes requisitos:
    • • Simplicidade - O usuário não quer perder tempo com uma interface complicada, que exija muitos passos para efetuar uma tarefa simples.
    • • Clareza - Temos sempre que ter em mente o seguinte princípio: “o usuário não quer – aliás, não deve – pensar”. Isso que dizer que o usuário vai se irritar se tiver de decifrar uma interface que tenha ícones estranhos ou termos ambíguos nos menus e botões.

Conceitos

Alguns elementos devem ser considerados no processo de produção de interfaces. Estes elementos são: usabilidade, consistência, navegabilidade, interatividade, clareza, flexibilidade, funcionalidade e legibilidade.

  • [Usabilidade] - Segundo a norma ISO 9241, é a capacidade que um sistema interativo oferece a seu usuário, em um determinado contexto de operação, para a realização de tarefas com qualidade e eficiência. Podemos considerar como uma propriedade que um aplicativo deve possuir para que os usuários possam utilizálo para realizar bem as tarefas que necessitam.
  • [Consistência] - Refere-se à coerência do uso dos itens de interface. Todos os elementos que possuem as mesmas funções devem apresentar as mesmas características gráficas e operacionais.
  • [Navegabilidade] - Écapacidade que a interface do aplicativo possui de facilitar ao usuário chegar ao seu destino da maneira mais eficiente possível.
  • [Interatividade] - São um conjunto de operações ou atividades que o usuário põe em prática com os objetos de tal modo a alterar o ambiente.
  • [Clareza] - Diz respeito à evidência de se encontrar os itens principais da interface, ou seja, eles devem ser claros o suficiente para que não haja dúvida sobre sua funcionalidade e seu uso. Se não são evidentes, devem ser autoexplicativos.
  • [Flexibilidade] - É a capacidade da interface de se adaptar as várias ações do usuário.
  • [Funcionalidade] - Consiste nos elementos da interface cumprirem com eficiência o papel que lhes cabe. Deve-se combinar a clareza da função proposta com a eficiência na tarefa de realizar esta função.
  • [Legibilidade] - Diz respeito às características lexicais das informações apresentadas na tela que possam dificultar ou facilitar a leitura dessa informação (brilho do caractere, contraste letra/fundo, tamanho da fonte, espaçamento entre palavras, espaçamento entre linhas, espaçamento de parágrafos, comprimento da linha, etc.).

Princípios

Alguns princípios devem ser levados em consideração no projeto da interface de uma aplicação. Veremos algumas características a seguir que servirão como base para os desenvolvedores no desenho da interface.

Principios básicos de usabilidade

O desenho da aplicação deve ser centrado no usuário, pois nem sempre o que é bom para o desenvolvedor é bom para o usuário.

  • Deve-se conhecer o comportamento, hábitos e habilidades do usuário.
  • O usuário deve participar da elaboração do aplicativo, testandoo e avaliando de acordo com suas necessidades.
  • Peça confirmação para ações de risco.
  • Ações que não forem pertinentes devem ficar indisponíveis para o usuário.
  • Procure otimizar operações para usuário, com uso de atalhos ou através de sequências de ações.
  • Ajude o usuário iniciarse no sistema, apresentando de forma intuitiva os recursos disponíveis.
  • Consistência e simplicidade, sempre!
  • Apresente recursos familiarizados pelo usuário e use (com cuidado) analogias do mundo real.
  • Apresente mensagens claras e positivas para o usuário.
  • Se possível, permita que o usuário possa reverter situações e ações.
  • O usuário espera e deseja consistência.
  • A tela deve ser simples. Apresentação de dados e objetos desnecessários, que dão a impressão de que a tela é uma grande aglomeração de informações, deve ser evitada.

Estrutura

O corpo da estrutura padrão das aplicações web está divido em 4 partes, sendo elas topo, barra de acessibilidade, menu de aplicação lateral e container para desenvolimento dos casos de uso.

A "estrutura padrão" está preparada para se adaptar em diversas resoluções. Deixamos claro que na construção deste Padrão de Interface cada elemento, cor, fonte e comportamento foram elaborados de forma a atender as necessidades do usuário final, por esse motivo, sugestões de alteração do mesmo devem ser aprovadas pela equipe responsável pela evolução do padrão.

Detalhamento da Estrutura
  • [A - Espaço destinado ao ícone de representação do sistema] - Elemento importante para identificação do sistema no Portal Goiás que deve ser definido junto ao gestor responsável pelo projeto.
  • [B - Espaço destinado ao nome do sistema] - Vale ressaltar que as características de fonte como tipo, cor e espessura não devem ser alteradas para atendimento ao padrão. Em cenários onde o nome do sistema venha ultrapassar o espaço destinado ao nome, um ajuste do tamanho da fonte poderá ser aplicado desde que homologado junto a equipe de testes.
  • [C - Espaço para marca da superintenência do sistema ou órgão responsável pelo sistema] - Como no modelo, esta marca necessariamente precisa ser branca com background transparente. Dimensões não podem ser alteradas.
  • [D - Espaço para marca do órgão de maior hierarquia ao qual o sistema é subornidado] - Como no modelo, esta marca necessariamente precisa ser branca com background transparente. Dimensões não podem ser alteradas.
  • [E - Espaço destinado a marca de representação do governo corrente] - Assim como as demais, é um elemento obrigatório e não deve ser alterado
  • [F - Label para indicação do tipo de tela da funcionalidade]
  • [G - O label de indicação de detalhe e agrupamento de elementos de formulário] - Tem como objetivo, apoiar na organização do forumlário, tornando mais claro e de fácil preencimento.
  • [H - Container para menu e ícones para identificação de acesso à funcionalidade] - Utilizar palavras curtas para representação da funcionalidade para melhor harmonia da coluna de ítens de menu.

Topo

Em resumo, esta é a visão do topo do layout do padrão de interface. Como detalhado no tópico anterior, estas são as únicas informações textuais e visuais que o topo dos sistemas aderentes a este padrão de interface deve conter exatamente com essas caracteristicas.

Barra de Ferramentas

Assim como os demais componentes que compõem a 'estrutura' deste padrão interface, a barra de ferramentas é um recurso de grande importância para usuário e para a qualidade do sistema. Dividida em três grupos de recursos, provê ferramentas de acessibilidade, botões para configuração, identificação de versão do sistema e função para encerrar a sessão do usuário.

O detalhamento de cada recurso da barra de acessibilidade pode ser na sessão Usabilidade

Menu Lateral

Menus são elementos de navegação da interface que devem apresentar as opções de navegação da aplicação. A escolha do uso de menus laterais permite uma maior visibilidade a todos os ítens de menu sem que os mesmos sobreponham a área útil dos formulários, facilitando assim ao usuário memorizar todas as operações disponíveis na aplicação. O usuário deve ter opção de expandir e recolher todos os ítens de menu da forma que mais lhe atenda.

Agrupe adequadamente todos os ítens de menu de acordo com sua função evite ao máximo criar subníveis. A nomenclatura do menu e do submenu deve ser bem clara para o que usuário possa identificar exatamente o que aquela função se propõe.

Utilize no máximo 3 níveis de menu para facilitar o acesso do usuário ao recurso desejado.

Container Principal

Este é o container principal da aplicação. Esta área é destinada as informações da aplicação, como por exemplo: listagem dos dados, cadastro de informações, relatórios.

O container principal deve sempre ser identificado e sempre que possível também deve ser dividido em sessões para melhor orientação do usuário.

Padrões de UI

Uma maneira eficaz de se documentar soluções recorrentes e consolidadas para resolução de problemas comuns e recorrentes de interface de usuário.

Se temos componentes com funções comuns em diferentes interfaces, esses elementos devem ser iguais (ou pelo menos o mais semelhante possível), tornando assim o processo de evolução da ui(correções ou melhoria), mais simples de ser executado.

Tela de Apresentação

A tela de apresentação padrão do sistema deve ser utilizada como ponto de acesso inicial de todas aplicações.

Informações como contato do responsável e ou suporte, nome e descrição do sistema são indispensáveis neste espaço.

Este espaço também pode ser utilizado para exibição de informativos e orientações sobre do mantidas para o sistema.

Layout de Formulário - Pesquisa

Os elementos de formulário, neste exemplo "os filtros", devem estar dispostos em duas colunas dentro do container de formulário. Isso proporcionará uma tela mais leve, com menos informações e de fácil interpretação para o usuário final. Esta disposição dos elementos de formulário também garante uma aplicação mais simplificada deste padrão de interface por ferramentas geradoras de códigos.

Detalhamento
  • [A - Coluna para label de ítem de formulário] - Neste caso, os labels que precedem elementos de formulário devem sempre estar alinhados à direita.
  • [B - Coluna de elementos de formulário] - Este mesmo alinhamento se aplica aos elementos de formulários.
  • [C - Área para descanso dos olhos] - O espaço poderia estar preenchido por uma disposição em linha dos ítens de formulários, porém o padrão duas colunas foi escolhido por atender de forma mais simples e de rápida implementação grande parte dos cenários.
  • [D - Botão padrão para adição de novos ítens] - Este botão de operação deve estar sempre à direita superior do container, logo após a área de descanso, faciliantado assim que o usuário encontre de forma mais clara o acesso à essa operação.
  • [E - Botões padrão para aplicação dos filtros e limpeza] - Estes botões devem sempre estar alinhados à base dos itens do formulário de filtros de pesquisa.

Crud Padrão 01 - Lista Paginada

Este é o Crud Padrão 01. Como primeira tela do fluxo do crud, basicamente é composto por uma lista paginada, campos de filtro para a lista paginada e um botão para acesso à tela de inclusão de dados, segunda tela do fluxo do crud padrão .

O detalhe mais importante desta tela, é o fato da lista paginada não ser carregada no load da tela. Por padrão, todas as listas paginadas deverão ser carregadas somente após o preenchimento dos filtros. Esta decisão tem como objetivo, evitar consultas ou pesquisas não solicitadas pelo usuário. Isso garantirá uma melhor utilização dos recursos dispostos para aplicação.

Detalhamento da Estrutura
  • A - Filtros para lista] - Em primeiro lugar, orientamos que cada elemento de formulário ou componente a ser utilizado como filtro de pesquisa deve ser obtido da biblioteca de componentes do "framework corporativo" da tecnologia em questão .Net, Java ou PhP, lembrando que os nossos frameworks corporativos estão em constante evolução, e na ausência de algum componente específico o mesmo poderá ser solicitado junto à equipe de arquitetura. Por padrão, todos os filtros devem iniciar vazios, para que apenas sejam preenchidos pós iterações do usuário com a tela. Observação: Não alterar qualquer característica dos labels ou elementos de formulário como por exemplo fonte, cor, expessura, posicionamento e etc.
  • [B - Botão para filtragem] - Dispara a função de pesquisa considerando os filtros preenchidos. Na ausência de registros, uma única linha na lista paginada deve conter a mensagem "Nenhum registro encontrado" que informa o usuário sobre o resultado da pesquisa.
  • [C - Botão para limpeza de elementos de formulário] - Deve sempre estar a direita de todos os demais botões do formulário exatamente com a estilização demonstrada no print.
  • D - Lista paginada] - A lista paginada possui papel importante no Crud Padrão 01, pois através dela o usuário poderá encontrar suas informações cadastradas para manutenção das mesmas. A lista paginada não deve ser carregada no load na tela para evitar consultas desnecessárias ao servidor, o carregamento da lista paginada só deve ser feito após o disparo da ação filtrar pelo botão filtrar.
  • [E - Lista paginada: Títulos de Colunas] - Devem sempre estar alinhados à esquerda.
  • [F - Label instrução para agrupamento] - Se trata de uma função específica agrupamento, abordada no tópico "Componentes"
  • [G - Controle de paginação padrão] - O controle de paginação das listas paginadas utilizados nos frameworks corporativos devem seguir este padrão independente do componente adotado, com as funções abaixo.
  • [G.1 - Acesso Rápido] - Por número de página
  • [G.2 - Configuração de Registros] - A serem exibidos por página
  • [G.3 - Paginador] - Botões para avançar ou retornar uma página.
  • [H - Mensagem da lista paginada] - No caso de não encontrar ítens para a pesquisa.
  • [I - Título da Funcionalidade] - Exemplificando o detalhamento do título abordado no tópico Estrutura
  • [J - Label de Identificação do Tipo de Tela] - Exemplificando o detalhamento do ítem (F) no tópico Estrutura

Crud Padrão 01.1 - Lista Paginada Preenchida

Esta é uma implementação da lista paginada com os recursos e comportamentos mínimos esperados.

Além do carregamento da lista somente após iteração do usuário com o botão filtrar*, deve se considerar que o carregamento dos itens da lista paginada não exceda 2 segundos.

Todas listas paginadas que estejam sob o padrão de interface Goiás devem possuir todas essas característas apresentadas no print e na detalhamento.


Detalhamento
  • [A - Lista paginada] Carregada com os parâmetros de pesquisa.
  • [B - Cores de Linha] Necessárias para análise dos dados por parte do usuário final.
  • [C - Coluna Ordenada] Ao ordenar uma coluna, é necessário que a coluna em ordenação utilize este background para identificação da ordenação aplicada. Observando que o identificação de linha par e ímpar para zebragem deve permanecer mesmo que em tom mais escuro.
  • [C.1 - Botão para ação de Ordenação]Ordena de forma ascendente ou descendente a coluna desejada além de também indicar quar ordação está aplicada. Em cenários onde a tela deva ser carregada com ordenação padrão o mesmo deve indicar tal ordenação padrão.
  • [D - Cor de Linha] por hover. Indica qual linha receberá o clique do mouse durante o evento hover.
  • [E - Reticências] Necessária e indispensável nos casos onde a lista paginada não suporta o redimensionamento de tela.
  • [F - Totalizador de registros encontrados]Deve ser atualizado sempre que a ação filtrar for executada.
  • [G - Filtro informado] Deve permanecer preenchido até que o mesmo venha ser limpo por ação do botão limpar, ou quando a tela for recarregada.

Fluxo 01 - Lista para Form :: Incluir

Este fluxo exemplifica a inclusão de um novo registro no Crud Padrão 01.

Este fluxo padrão deve ser adotado em todos os cruds básicos da aplicação. O mesmo é divido basicamente em duas telas, sendo a primeira o "Crud Padrão 01" e a segunda pelo "Crud Padrão 02". O fluxo é dividido em duas fases para simplificação do desenvolvimento bem como execução de testes e entregas de demandas.

Detalhamento
  • [A - Novo Item] - Neste fluxo e nesta tela de crud, tem a única função de encaminhar o usuário para a tela de entrada de dados.
  • [B - Campos de Formulário] - Elementos de formulários para entrada de dados pelo usuário. Comportamentos e funções de cada componente corporativo de formulário exemplo input text, select, textarea e etc são detalhados na sessão "Componentes".
  • [C - Salvar] - Botão para enviar os dados ao back-end para validação e persistência dos dados. Todas as mensagens de negócio retornadas devem ser apresentadas nesta mesma página.
  • [D - Mensagem Sucesso] - Mensagem de sucesso retornada pela operação de "inclusão" de dados. Toda operação disparada pelo usuário através dos botões de controle do formulário, devem retornar uma mensagem sobre o resultado da mesma, seja sucesso ou não. Para informações e detalhamento do padrão de mensagens, consulte o tópico "Validações de Regras".
  • [E - Atualização de Tela] - Ao realizar a "operação" com sucesso, a aplicação deve retornar a tela a qual o usuário iniciou sua solicitação de inclusão de dados. E esta tela deve estar em seu estado inicial, sem a seleção qualquer filtros ou dado carregado na lista paginada inicial.
  • [F - Evidenciação de Operação Realizada] - Caso o usuário necessite verificar a informação incluída, o mesmo precisará efetuar uma pesquisa ou consulta informando os dados desejados. Desta forma, mantemos a política de somente oferecer ao usuário, informações que o mesmo solicitou.

Fluxo 01.1 - Lista para Form:: Incluir :: Validar

Este fluxo exemplifica o comportamento e tratamento de tela de crud após tentativa de inclusão de novo registro sem que os campos obrigatórios sejam informados.

Detalhamento
  • [A - Salvar] - Neste caso, se trata de uma operção de inclusão.
  • [B - Obrigatoriedade] - Informa os campos obrigatórios para o crud.
  • [C - Mensagem de Erro] - Neste caso, se trata de uma mensagem padrão que informa ao usuário a existência de inconsistências que impedem que sua operação tenha prosseguimento.
  • [D - Mensagem de Erro por Elemento] - Todo campo obrigatório, necessariamente deve receber sua mensagem de erro específica, podendo ser uma obrigatoriedade de campo, campo informado incorretamente ou condição de preenchimento.

Fluxo 02.1 - Lista para Form:: Editar

Este fluxo padrão 02, define comportamento e tratamento de tela para alteração de dados por meio do mesmo formulário utilizado no fluxo 01 para entrada de novos dados.

O fluxo se resume em:

- O usuário seleciona a funcionlidade desejada através o menu da aplicação e recebe o Crud Padrão 01 para inicar este fluxo que detalhamos abaixo.

Detalhamento
  • [A - Filtro] - Neste caso, apenas um filtro para pesquisa, isso permite uma perquisa mais acertiva. Porém, caso o usuário não informe nenhum item, o sistema dele lhe entregar a primeira página da listagem dos dados da funcionalidade em questão.
  • [B - Filtrar] - Dispara a pesquisa com o filtro informado.
  • [C - Resultado] - Caso encontre algum resultado para a pesquisa solicitada, a lista paginada é atualizada com os itens encontrados.
  • [D - Foco em Linha da Tabela] - Todas os itens de listas devem possuir este background aplicado pelo hover do mouse. Isso facilita a análise dos dados por parte do usuário final para que o mesmo possa selecionar o elemento da lista paginada de forma mais clara. Neste caso, o clique na linha iluminada encaminhará o usuário para tela de inclusão de dados, que neste caso terá as funções de alteração ou exclusão disponíveis dependendo do nível de acesso que o usuário possuir.
  • [E - Entrada de Dados] - Neste caso, exemplificamos a atualização do dado para que o mesmo seja enviado para persistência.
  • [F - Botão Salvar] - Neste caso terá função de enviar os dados para que a aplicação aplique as alterações efetuadas pelo usuário final no registro consultado.
  • [G - Mensagem Retornada ao Usuário] - Em caso de sucesso, a mensagem deve informar ao usuário sobre o sucesso na alteração/atualização dos dados informados.
  • [H - Label Indicativo de Operação] - Como detalhado no tópico Estrutura o label deve informar o usuário sobre
  • [I - Tipo de Tela/View] - Alterado de acordo com os padrões definidos no tópico Estrutura.

Fluxo 02.1 - Lista para Form:: Excluir

Este fluxo exemplifica a exclusão de um registro utilizando-se o Crud Padrão 02.

Detalhamento
  • [ A - Filtro para Pesquisa] - Usuário informa o filtro de pesquisa para obter o registro para exclusão. Neste ponto, vale lembrar que usuário pode a qualquer momento disparar a pesquisa da lista paginda sem informar os filtros, assim ele obterá a primeira página da lista de registros e dessa forma poderá navegar na lista paginada até encontrar o resultado esperado.
  • [ B - Botão para Filtragem] - Obtém a primeira página da lista paginda com base nos filtros, ou a primeira página da listagem sem filtros caso não sejam informados.
  • [ C - Resultado] - Registro(s) encontrado(s).
  • [ D - Foco em Linha para Exclusão] - Iluminação de linha para apoiar a seleção acertiva do ítem para exclusão.
  • [ E - Preenchimento de Campos] - Os campos devem ser preenchidos para que o usuário tenha certeza de que os dados a serem excluídos estejam de acordo com sua intenção.
  • [ F - Botão para Excluir] - Este botão somente deve ser visível ao usuário somente quando as informações preenchidas no formulário se tratarem de informações já persistidas pela aplição, o que não acontece no detalhamento do Crud Padrão 02.1.
  • [ G - Modal de Confirmação] - O modal de confirmação funciona como uma estratégia de "segurança" a fim de se ter uma garantia "mínima" da inteção de exclusão do registro por parte do usuário, evitando assim um prejuízo ao usuário por uma ação ocasionada por um engano. O mesmo também pode ser utilizado antes de qualquer outro tipo operação que se queira avisar o usuário sobre a indisponibilidade de reverter a ação.
  • [ H - Mensaegm de Confirmação] - Neste caso a mensagem "padrão" para solicitar a confirmação da intenção do usuário em excluir o registro.
  • [ I - Botão para Confirmar Exclusão*] - Este é o botão padrão para efetivar a exclusão.
  • [ J - Botão para Cancelar a Intenção de Operação] - Neste caso, este botão funciona como recurso para que o usuário tenha opção de desistir de efetuar a operação. Além de cancelar a inteção de exclusão do registro, também tem função de fechar o modal e habilitar a tela sobreposta pelo modal.
  • [ J.1 - Botão para Cancelar a Intenção de Operação] - Tem a mesma função do item (J), além de cancelar a inteção de exclusão do registro, também tem função de fechar o modal e habilitar a tela sobreposta pelo modal.
  • [ K - Background de Bloqueio] - O background de bloqueio de tela para obtenção de confirmação de operação.
  • [ L - Mensagem Retornada ao Usuário] - Em caso de sucesso, a mensagem deve informar ao usuário sobre o sucesso na exclusão do registro selecionado.

Formulários

Quando se trata da formatação de formulários, é importante ter em mente algumas considerações.

  • Não mude a forma básica dos elementos. Quando o usuário se depara com um formulário com um formato muito diferente do acostumado, ele pode se sentir constrangido e simplesmente abandonar o preechimento.
  • Utilize rótulos (labels) claros e curtos. Tente ser conciso na nomenclatura dos campos.
  • Campos maiores, respostas maiores. Procure deixar o tamanho dos campos proporcional ao tamanho da informação que o sistema espera receber. Se um campo exige uma informação curta, use campos curtos. Por exemplo, não coloque um campo senha com tamanho de 300px se espera receber apenas 6 caracteres. O usuário certamente vai pensar que não entendeu bem a finalidade do campo.
  • Sinais de identificação. É interessante usarmos * para um campo obrigatório. O usuário está acostumado com esse sinal. Iremos utilizálo antes do campo do formulário, a direita do rótulo. É preciso que o usuário saiba que aquele campo é obrigatório antes de ler o campo.
  • Agrupe. Procure agrupar informações semelhantes e complementares.
